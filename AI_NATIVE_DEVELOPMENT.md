# AI-Native Development Standard

This document defines the AI-native development philosophy, workflow, and standards
for the ComplyTime organization. It is the single source of truth for how AI tools
are used across all repositories.

For per-repo setup and operational guidance, see `docs/AI.md` in each repository.

---

## Philosophy

AI-native development treats AI agents as first-class collaborators in the software
development lifecycle. Agents assist with specification, implementation, and review
-- but human judgment remains the authority on all merge decisions.

Three principles govern this approach:

1. **Spec-driven development**: Every feature starts with a specification that
   defines WHAT to build before HOW to build it. This anchors both human and AI
   work to a shared intent.
2. **Convention-pack enforcement**: Coding standards are encoded as convention packs
   that agents read and enforce during implementation and review. This ensures
   consistent quality regardless of which contributor or agent does the work.
3. **Council review**: Multiple AI review personas examine every PR from different
   perspectives (security, architecture, testing, operations, governance). This
   provides breadth of analysis that complements human depth.

---

## Workflow

The AI-native development workflow consists of five steps:

### 1. Specify

Define what to build using a spec-driven framework (SpecKit or OpenSpec). The
specification captures requirements, acceptance scenarios, and design decisions
before any code is written.

- **Input**: Feature idea, bug report, or enhancement request
- **Output**: Spec artifacts (proposal, design, specs, tasks)
- **Who**: Contributor + AI agent
- **Tools**: `/speckit.specify`, `/opsx-propose`

### 2. Spec Review

The review council and human maintainers review the specification for intent
fidelity, architectural soundness, security gaps, testability, and operational
feasibility.

- **Input**: Spec artifacts
- **Output**: Approved spec (or findings to address)
- **Who**: AI council (5 Divisor personas) + human maintainers
- **Tools**: CI council review workflow (automatic), `/review-council` (local)

### 3. Implement

Build the feature following the approved specification. Convention packs guide
the agent's coding decisions. Tasks from the spec are executed sequentially.

- **Input**: Approved spec + convention packs + constitution
- **Output**: Working code with tests
- **Who**: Contributor + AI agent
- **Tools**: `/opsx-apply`, `/unleash`

### 4. Code Review

The review council and human maintainers review the implementation for
constitution compliance, security, test quality, performance, and spec alignment.

- **Input**: Implementation code (PR diff)
- **Output**: Approved code (or findings to address)
- **Who**: AI council (5 Divisor personas) + human maintainers
- **Tools**: CI council review workflow (automatic), `/review-council` (local)

### 5. Merge

Human maintainers make the final merge decision. AI findings inform but do not
replace human judgment.

- **Input**: Approved code + passing CI
- **Output**: Merged PR
- **Who**: Human maintainers (minimum 2 reviewers)

## File Classification

AI-native development introduces files that fall into two categories. Understanding
which files are committed and which are local-only is essential for consistent
collaboration.

### Committed files (tracked in git)

These files ensure all contributors and agents follow the same standards regardless
of their tool choice.

| File | Purpose | Managed by |
|------|---------|------------|
| `.specify/memory/constitution.md` | Governance and coding standards | org-infra (synced) |
| `.opencode/uf/packs/*.md` | Convention packs (coding rules) | Committed after `uf init` |
| `.opencode/agents/divisor-*.md` | Divisor review personas (5 core) | org-infra (synced) |
| `.opencode/commands/review-pr.md` | PR review command | org-infra (synced) |
| `docs/AI.md` | Per-repo AI development guide | org-infra (synced) |
| `AGENTS.md` | Project structure and build commands | Repo maintainers |
| `specs/`, `openspec/` | Feature specifications | Contributors |

### Local-only files (not committed)

These files depend on the contributor's agent choice, plugin version, or personal
preferences. They are generated by tools and should not be committed.

| File | Purpose | Generated by |
|------|---------|-------------|
| `.opencode/commands/speckit.*.md` | SpecKit framework commands | Plugin install |
| `.opencode/commands/opsx-*.md` | OpenSpec framework commands | Plugin install |
| `.opencode/agents/divisor-{curator,scribe,herald,envoy}.md` | Content personas | `uf init --divisor` |
| `.opencode/commands/review-council.md` | Local review council command | `uf init --divisor` |
| `.cursor/`, `.vscode/` | Editor-specific AI configs | Editor |
| MCP server configs | Tool-specific connectivity | Contributor |

## Convention Pack System

Convention packs encode the coding rules that agents enforce during implementation
and review. They are the bridge between governance (the constitution) and
day-to-day coding.

### Hierarchy

When multiple sources define the same rule, the more specific source wins. Custom
packs MAY tighten rules (SHOULD to MUST) but MUST NOT relax them (MUST to SHOULD).

```text
Project custom pack    (.opencode/uf/packs/<lang>-custom.md)   Wins
  |  extends
Generic language pack  (.opencode/uf/packs/<lang>.md)          Baseline
  |  must not contradict
Constitution           (.specify/memory/constitution.md)       Authority
```

### How packs work

1. **Generic packs** (e.g., `go.md`, `default.md`) are maintained in
   [unbound-force](https://github.com/unbound-force/unbound-force) and scaffolded
   into repositories via `uf init`.
2. **Custom packs** (e.g., `go-custom.md`, `default-custom.md`) are maintained by
   each repository's maintainers. They add project-specific rules.
3. Agents MUST read the applicable convention packs before writing or reviewing code.
4. Each rule in a pack has a severity (`[MUST]`, `[SHOULD]`) and a rationale.

### Where to capture improvements

When a pattern or rule needs to be added:

| Pattern scope | Where to add it |
|---------------|----------------|
| This repository only | Project custom pack (`*-custom.md`) |
| All repositories in the org | Generic pack (propose to unbound-force) |
| Governance-level rule | Constitution (propose to org-infra) |

## Disclosure Standards

### Assisted-by trailer

All commits that were authored or substantially assisted by an AI tool MUST include
an `Assisted-by` trailer identifying the tool and model used:

```text
feat: add validation endpoint

Implement input validation with JSON schema support.

Signed-off-by: Jane Doe <jane@example.com>
Assisted-by: OpenCode (claude-opus-4-6)
```

This ensures transparency and traceability of AI-assisted contributions.

### Agent-agnostic principle

Agent choice is personal. Contributors may use OpenCode, Claude Code, Cursor, or
any other AI agent that supports command loading. The committed files (constitution,
convention packs, AGENTS.md, specs) ensure consistent outcomes regardless of which
agent a contributor uses.

What matters is the output, not the tool:
- Code MUST comply with the constitution and convention packs
- Commits MUST include the `Assisted-by` trailer
- Specs MUST follow the framework conventions (SpecKit or OpenSpec)

## Continuous Improvement

The AI-native workflow includes a feedback loop for improving standards over time.

### How improvements are captured

When the review council identifies recurring patterns, or a contributor discovers
a better practice, the improvement is captured in the appropriate location:

1. **Review council finds a recurring issue** (e.g., "contributors keep forgetting
   to handle context cancellation in Go"): Add a rule to the project custom pack
   or propose it for the generic pack if it applies org-wide.
2. **A convention pack rule is too strict or too lenient**: Adjust the severity
   (`[MUST]` vs `[SHOULD]`) in the custom pack, or propose the change to the
   generic pack.
3. **A governance gap is discovered** (e.g., no standard for container attestations):
   Propose an amendment to the constitution via org-infra PR.

### Feedback sources

- **Council review findings**: Persistent findings across PRs indicate missing rules
- **Retrospectives**: The `/unleash` pipeline captures learnings after each feature
- **Human review**: Maintainer observations during PR review

## Quick Reference

### For contributors

| Task | How |
|------|-----|
| Start a new feature | `/speckit.specify` or `/opsx-propose` |
| Implement a feature | `/opsx-apply` or `/unleash` |
| Review a PR locally | `/review-pr <number>` |
| Run the review council | `/review-council` (local, via `uf init`) |
| Check coding standards | Read `.specify/memory/constitution.md` |
| Check convention packs | Read `.opencode/uf/packs/*.md` |
| Set up AI tooling | Read `docs/AI.md` in the repository |

### For maintainers

| Task | How |
|------|-----|
| Update convention packs | Edit `*-custom.md` packs, commit |
| Sync AI tooling | org-infra `sync-config.yml` manages distribution |
| Scaffold a new repo | `uf init --divisor --lang <lang>` |
| Propose governance changes | PR to org-infra `constitution.md` |
